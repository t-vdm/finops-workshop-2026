FOCUS

You already export to FOCUS.

What FOCUS gives you, that raw exports don't

  • One schema across Azure, AWS, GCP, Oracle, Databricks, Snowflake. Portable mental model, portable dashboards.
  • Stable column names. Microsoft can change EA → MCA → CSP underneath; FOCUS abstractions don't break your reports.
  • Four cost flavors, not one: Billed, Effective, List, Contracted. Each answers a different question.
  • Industry-standard semantics. Hire a FinOps practitioner and they're productive on day one.

Where the toolkit stops

Where the Toolkit stops, and DIY begins

Toolkit gives you

  • FOCUS-shaped cost data in your own storage
  • Reservation utilisation snapshots
  • Advisor recommendations passthrough
  • Pre-built PBI reports + workbooks
  • Open data for service-name normalisation

Toolkit doesn't give you

  • Live Azure Resource Graph joins (config, tags-at-rest)
  • Per-VM rightsizing with CPU/memory metrics
  • Cross-tool prioritisation (cost + security + ops)
  • Forecasting + anomaly detection out of the box
  • Multi-tenant slicing (relevant if you have BUs or M&A activity)
  • EOL/retirement-driven cost forecasting
  • "Saved scopes / lenses" for different audiences
Tagging

Most FinOps efforts die here.

30–50%
Of cloud spend untagged or inconsistently tagged at most enterprises (FinOps Foundation surveys)
>90%
Tag compliance is the initial goal per the FinOps Tagging WG. Harder than it sounds.
~12mo
Typical time to drag a brownfield Azure estate to that goal without enforcement automation

Failure modes you'll recognise

  • A
    Tags exist on Resource Groups but not children. Azure does not auto-inherit. Most "untagged" rows live here.
  • B
    costCenter, CostCenter, cost-center, cc all coexist by week two. Case + alias sprawl.
  • C
    Tag-at-deploy works; tag-at-modify is forgotten. Drift is the silent killer.
  • D
    "Untagged" treated as a backlog ticket rather than a tracked, time-series KPI on an executive dashboard.
Tagging · what works

A working strategy

You've already had separate sessions on the taxonomy itself, so we won't relitigate. A few reinforcements worth locking in around it.

1
Define mandatory taxonomy
Before anyone deploys. Done. Keep iterating quarterly with finance.
2
Enforce via Azure Policy
deny on missing tags · modify to inherit RG → child resources.
3
Parse a defined set · ignore the noise
Only consume the tags you defined. Auto-assigned tags from Databricks, AKS / Kubernetes, Veeam, ARM templates and platform agents are noise. They pollute reports and break allocation.
4
Normalise at ingest
Case-fold + collapse aliases (costCenter = CostCenter = cc). Tag-lint pipeline before the data lands in your warehouse.
5
Track coverage as a KPI
Untagged % over time, on an executive dashboard. Not in a backlog ticket.
6
Allocate the unallocatable
Cost Management allocation rules for shared infra · fallback by sub owner so nothing lands in "unowned".

Allocation · the shared-cost problem

Some costs can't be tagged.
That's what allocation rules are for.

The unallocatable layer

Hub VNets · ExpressRoute · Log Analytics workspaces · shared identity · monitoring · platform DevOps

You can't put one CostCenter tag on these. They serve everyone.

Four solutions, in order of maturity

  • Cost Management allocation rules: split by usage proxy (e.g. tagged-resource %)
  • Showback before chargeback. Surface allocations for 1–2 quarters before invoicing internally.
  • Azure's x_CostCenter column. Separate from Tags, set by billing admin.
  • Default fallback: x_AccountOwnerId / sub owner. Nothing should land in "unowned".

Operational gotchas

Operational gotchas

  • 1
    Cost Mgmt rate limits are tighter than docs claim. Reality: 10/min/tenant + 4/min/scope. Build a token-bucket scheduler, not a naive loop.
  • 2
    Resource Graph throttling. Query across subscriptions in one KQL; partition results client-side. One query per sub = 429.
  • 3
    Volume kills small containers. Multi-GB FOCUS exports. Stream Parquet to disk, transform lazy, sync in chunks.
  • 4
    EA vs MCA vs CSP. Different scope hierarchies. Use FOCUS abstractions; resist hardcoding.
  • 5
    Currency / FX. x_BillingExchangeRate × x_BillingExchangeRateDate. Don't sum raw BilledCost across rate changes.
  • 6
    Actual vs amortized cost. Pick a default (Effective for trends, Billed for finance). Make the toggle explicit in the UI.
  • 7
    Refresh latency. Cost Mgmt data is 8–24h delayed. Always show "as of <timestamp>".
  • 8
    Tag inheritance. Children don't inherit RG tags. Either Policy modify, or join through ARG to fill blanks at query time.
  • 9
    Untagged ≠ unallocatable. Default fallback by x_AccountOwnerId. Make untagged % a tracked KPI.
  • 10
    Schema drift. FOCUS evolves (1.0 → 1.0r2 → 1.1 → 1.2). Pin a version, validate on ingest, alert on shape change.

Partitioning + retention

Monthly partitions on charge_period_start. 13 months hot, archive to cold storage after. Indexes on (sub_account_id, charge_period_start) and (tag_costcenter, charge_period_start).

The Tags column problem

Three ways to handle Tags. Pick a hybrid.

Query JSON inline

Tags->>'CostCenter'

Simple

No transform step. Works on day one.

But: slow at scale, no index unless you create expression indexes per key.

Long-format tag table

(cost_id, key, value)

Generic

Index on (key, value). Any tag becomes filterable.

But: cardinality explodes, joins everywhere, harder for BI tools.

Pivot mandatory keys to columns

tag_costcenter, tag_environment, ...

Recommended hybrid

Mandatory keys → real columns (fast, indexed). Everything else stays in tags_raw json.

Best of both. Fast for the questions you ask 95% of the time.

Pivot SQL at ingest

SELECT
  resource_id,
  charge_period_start,
  effective_cost,
  tags->>'CostCenter'  AS tag_costcenter,
  tags->>'Environment' AS tag_environment,
  tags->>'Application' AS tag_application,
  tags->>'Owner'       AS tag_owner,
  tags                  AS tags_raw
FROM stg_focus_cost
WHERE charge_period_start >= :ingest_window;
Pre-aggregation patterns

Common rollups, illustrated in SQL

SQL shown is illustrative (Postgres flavour). The same patterns apply as Fabric semantic models, ADX summarize queries, or Power BI aggregations — the engine changes, the shape doesn't.

Monthly by CostCenter

The chargeback view. Refresh daily, full rebuild monthly.

CREATE MATERIALIZED VIEW
  mv_cost_monthly_by_cc AS
SELECT
  date_trunc('month', charge_period_start)
                          AS month,
  coalesce(tag_costcenter,'untagged')
                          AS cost_center,
  sum(effective_cost)     AS cost
FROM fact_focus_cost
WHERE charge_category = 'Usage'
GROUP BY 1, 2;

Daily by Service

Drives "what changed yesterday?" alerts.

CREATE MATERIALIZED VIEW
  mv_cost_daily_by_service AS
SELECT
  charge_period_start     AS day,
  service_category,
  service_name,
  sum(effective_cost)     AS cost
FROM fact_focus_cost
WHERE charge_category = 'Usage'
GROUP BY 1, 2, 3;

Reservation coverage

RI/SP utilisation tracking, monthly.

CREATE MATERIALIZED VIEW
  mv_ri_coverage_monthly AS
SELECT
  date_trunc('month', day) AS month,
  commitment_id,
  avg(utilization_pct)     AS avg_util,
  min(utilization_pct)     AS min_util
FROM fact_reservation_utilization
GROUP BY 1, 2;

Honest pick

  • Fabric Lakehouse if your data strategy is already Fabric-aligned. Easiest integration with Toolkit's Hub deployment.
  • ADX if dashboard latency matters more than tooling familiarity, and your team can adopt KQL.
  • Postgres + Timescale if you want simplicity, full SQL, and lower running cost. Scales fine for a single tenant.
Schema evolution

FOCUS will keep moving. Plan for it.

  • 1
    Pin the version on export. 1.0r2 today is stable; 1.2 once your downstream is ready. Don't auto-upgrade — schedule it.
  • 2
    Validate column shape on ingest. A 5-line check: assert the columns you depend on are present. Alert when Microsoft adds, renames, or removes a column.
  • 3
    Use a view layer between fact table and dashboards. Column renames (e.g. x_InvoiceId → InvoiceId from 1.0 to 1.2) get absorbed in one place instead of breaking 12 reports.
  • 4
    Backfill via Microsoft's 7-year re-export when bumping FOCUS versions. You can re-export historical data in the new shape — no need to migrate existing rows.
  • 5
    Track FOCUS conformance via Microsoft's published Conformance Report. Tells you which gaps still exist (e.g. CapacityReservation columns currently null).
